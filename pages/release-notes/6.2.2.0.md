Release date: **2026-02-22**

- Update account feature integrations aligned to latest FedNow/RTP model.
- Migrate `isAchPositivePay` handling from Legal Entity to Bank Account level.

## Detailed Changes

### LEI Data Validation via Open API Integration
- **What changed:** LEI validation is enforced across Customer Create/Update/Search APIs.
- **Why it matters:** Improves compliance with ISO 17442, prevents duplicates, and improves data quality.
- **Notes:** Backward compatible; LEI remains optional where applicable.

### Introduction of RTP-RFP in API with Payments
- **What changed:** Added support for RTP/FedNow RFP and RFI workflows with approval/rejection and inquiry endpoints.
- **Why it matters:** Enables end-to-end instant payment request workflows through API.
- **Notes:** No migration action required.

### FedNow and RTP Payment Creation via API
- **What changed:** Payment creation validates account-level Send and RFP Send flags correctly.
- **Why it matters:** Prevents false validation failures and aligns API behavior with account configuration.

### ACH Positive Pay Changes
- **What changed:** `isAchPositivePay` moved from Legal Entity to Bank Account.
- **Why it matters:** Positive Pay is now correctly modeled at account level.
- **Migration:** Update consumers to read/write this field from account-level payloads.

### Positive Rules for Instant Payments (RTP and FedNow)
- **What changed:** RTP and FedNow are grouped as a unified Instant category for positive rules.
- **Why it matters:** Simplifies rule setup and produces consistent rule evaluation.

### Positive Pay Exception Search Multi-Select Decision Type
- **What changed:** Exception search supports multi-select decision type filters.
- **Why it matters:** Improves precision and flexibility in operational searches.

### Remittance Information for SWIFT Payments
- **What changed:** SWIFT remittance info submitted via OpenAPI is captured and mapped consistently.
- **Why it matters:** Improves remittance visibility and downstream reconciliation.

### Unified Payment Rail and Account Feature Validation for ACH
- **What changed:** ACH rail/account feature validation now follows same framework as FedNow/RTP/Fedwire.
- **Why it matters:** Consistent validation behavior across all rails.

### Account Feature Changes (API)
- **What changed:** Deprecated fields removed and selected fields renamed for consistency.
- **Why it matters:** Aligns API model with current FedNow/RTP expectations.
- **Migration:** Update integration mappings to new field names and remove deleted-field references.

### Payment Webhook - API Specification Update
Webhook specification is aligned with actual payload for `paymentReturnDetails`.

Technical detail for `returnDate`:
- Proper naming (`returnDate`)
- Correct data type (`string` with `date` format)
- Expected format (`yyyy-mm-dd`)

### Capture and Transmission of Creditor Agent Account Type Information
- **What changed:** Added support for creditor agent account type capture/validation/transmission.
- **Why it matters:** Improves Fedwire ISO compliance and message fidelity.

### Incorrect Error Message When Accessing Unauthorized Payment
- **What changed:** Unauthorized access now consistently returns `Payment not found` with `API900`.
- **Why it matters:** Standardizes client-side error handling.

### Cancel Payment from API throws an Error
- **What changed:** Fixed cancellation authorization and payment state checks.
- **Why it matters:** Prevents false authorization failures and improves cancellation reliability.

### Incorrect Payment Creation Date and Time
- **What changed:** Corrected timestamp timezone handling in payment creation metadata.
- **Why it matters:** Prevents display/time-shift inconsistencies in downstream systems.
